# تقرير بنية قاعدة البيانات الشاملة (Database Schema Report)
## نظام دار شمس التعافي (Shams Recovery System)

هذا التقرير يوثق هيكلة قاعدة البيانات النهائية الجاهزة للعمل في الإنتاج (Production)، والتي تربط كل من Dashboard، Family Web Portal، و Flutter App.

---

## 1. الجداول الأساسية والعلاقات

تم تصميم الجداول لضمان الأمان، السرعة، والفصل التام بين بيانات كل مقيم.

### الجداول الإدارية:
- **`profiles`**: يربط مع `auth.users` ويحتوي على بيانات المستخدم وأدواره (`super_admin`, `admin`, `staff`, `doctor`, `therapist`, `family`).
- **`branches` و `rooms`**: لإدارة أفرع وغرف المركز.
- **`settings`**: لتخزين الإعدادات العامة للنظام بصيغة `JSONB`.

### الجداول التشغيلية (الطبية والإقامة):
- **`residents`**: الجدول المحوري. يضم بيانات المقيم، المرحلة العلاجية، والحالة، ويرتبط بأطباء ومعالجين من `profiles`.
- **`resident_updates`**: التحديثات اليومية واليوميات للمقيمين.
- **`weekly_reports`**: التقارير الدورية (أسبوعية/شهرية).
- **`sessions_attendance`**: متابعة الحضور في الجلسات الجماعية/الفردية.

### جداول التواصل (بوابة الأهالي والتطبيق):
- **`family_links`**: جدول وسيط يربط `profiles` (الأسرة) بـ `residents` (المقيم). يحتوي على **صلاحيات مخصصة** (عرض الصور، التقارير، إرسال الرسائل).
- **`messages`**: الرسائل المتبادلة بين الأسرة والإدارة، تدعم الردود `parent_message_id`.
- **`notifications`**: الإشعارات الموجهة للمستخدمين.
- **`news`**: الأخبار العامة، مع تحديد ظهورها في البوابة أو التطبيق.
- **`gallery`**: معرض الصور، ينقسم إلى `public` (مرافق) و `private` (مرتبط بمقيم معين).

### جداول النظام:
- **`file_number_sequences`**: جدول لمتابعة التسلسل التلقائي لأرقام الملفات (Sequence).
- **`audit_logs`**: يسجل جميع العمليات الحساسة (Insert, Update, Delete) مع حفظ `old_data` و `new_data`.

---

## 2. ميزات قاعدة البيانات المتقدمة

### رقم الملف التلقائي (Auto File Number)
تم برمجة `Function` + `Trigger` لتوليد رقم ملف حصري بصيغة `SHAMS-YYYY-XXXX` (مثل: `SHAMS-2026-0001`).
- يتم استخدام جدول `file_number_sequences` مع **Upsert** وتحديث القيمة لضمان عدم حدوث تكرار (Race Condition) حتى لو تم مسح مقيم.

### التحديث التلقائي (`updated_at`)
توجد `Function` مخصصة تقوم بتحديث عمود `updated_at` إلى الوقت الحالي عند أي عملية `UPDATE` في الجداول الهامة لتتبع التعديلات.

### سجل التدقيق (Audit Logs)
يتم تسجيل العمليات أوتوماتيكيًا عبر `Trigger` يسجل رقم حساب المستخدم، الجدول، نوع العملية (`INSERT`/`UPDATE`/`DELETE`)، ومحتوى البيانات في `audit_logs`.

---

## 3. أمن البيانات (Row Level Security - RLS)

تم تفعيل RLS على **كافة الجداول**، مما يضمن استحالة وصول بيانات مريض لعائلة مريض آخر:

- **الإدارة والطاقم الطبي**: يمكنهم قراءة وتعديل البيانات بناءً على الصلاحيات (`auth_user_has_role()`). 
- **الأسرة (Family)**:
  - لا يمكنهم رؤية المقيمين إلا المرتبطين بهم عبر `family_links` حيث `is_active = true`.
  - لا يرون `weekly_reports` أو `resident_updates` إلا إذا تم تعيين `visible_to_family = true` (وبالنسبة للتقارير يجب أن تكون `report_status = 'published'`).
  - في `messages`، لا يمكنهم القراءة أو الإرسال إلا لحسابهم الشخصي والمقيمين المرتبطين بهم إذا كانت `can_send_messages = true`.

---

## 4. مساحات التخزين (Storage Buckets)

تم تجهيز SQL لإنشاء وتأمين مساحات التخزين عبر Supabase Storage:
- **`public-gallery`**: (عام) للصور التسويقية وصور المرافق.
- **`news-images`**: (عام) لصور الأخبار.
- **`private-resident-media`**: (خاص) لا يمكن قراءته إلا للإدارة، أو الأسرة إذا كانت الصورة تابعة للمقيم المرتبط بهم ولديهم صلاحية `can_view_photos`.
- **`report-files`**: (خاص) لملفات PDF للتقارير الطبية.

---

## 5. طريقة ربط الأنظمة (Integration)

- **Next.js Dashboard**: يقرأ ويكتب من قاعدة البيانات مستخدمًا حساب موظف/أدمن (`role = admin/staff`)، فيتمكن من استعراض كافة البيانات والتحكم في ظهورها للعامة.
- **Next.js Family Portal**: يدخل الأسرة بنفس قاعدة البيانات (`role = family`)، ويقرأون فقط البيانات التي تسمح بها الـ RLS Policies. لا توجد حاجة لكتابة API معقد لأن Supabase يفلتر البيانات أوتوماتيكيًا.
- **Flutter App**: يعمل بنفس آلية الـ Family Portal تماماً مستخدماً نفس الـ `anon_key` وحساب الأسرة، ويستفيد من نفس الـ RLS.

---

## 6. تعليمات التنفيذ (Deployment)

1. افتح مشروعك على **Supabase**.
2. انتقل إلى قائمة **SQL Editor**.
3. قم بإنشاء Query جديد وانسخ محتوى ملف `supabase/schema.sql` بالكامل.
4. اضغط على زر **RUN**.
5. تأكد من عدم وجود أي أخطاء باللون الأحمر (تم استخدام `IF NOT EXISTS` لتجنب الأخطاء).
6. قاعدة بياناتك الآن جاهزة بجميع الجداول، العلاقات، والصلاحيات، ومستعدة للإنتاج (Production).
